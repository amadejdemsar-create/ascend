Create `lib/services/todo-service.ts` following the exact pattern of `goal-service.ts`:

```typescript
import { prisma } from "@/lib/db";
import type { CreateTodoInput, UpdateTodoInput, TodoFilters } from "@/lib/validations";
import { gamificationService } from "@/lib/services/gamification-service";
import { XP_PER_TODO } from "@/lib/constants";
```

Export a `todoService` object with these methods:

**`list(userId, filters?, pagination?)`**
- Query `prisma.todo.findMany` with where clause built from filters
- Filter by status, priority, categoryId, goalId (all exact match)
- Filter by dateFrom/dateTo using dueDate range: `dueDate: { gte: new Date(dateFrom), lte: new Date(dateTo) }`
- Filter by isBig3 if provided (convert string "true"/"false" to boolean)
- Order by `[{ sortOrder: "asc" }, { createdAt: "desc" }]`
- Include `category: true` and `goal: { select: { id: true, title: true, progress: true, targetValue: true } }`
- Support skip/take pagination

**`create(userId, data: CreateTodoInput)`**
- Validate goalId exists and belongs to user if provided (fetch from prisma.goal.findFirst)
- Validate categoryId exists and belongs to user if provided
- Create via prisma.todo.create with userId, converting dueDate and scheduledDate strings to Date objects
- Return the created todo

**`getById(userId, id)`**
- prisma.todo.findFirst where { id, userId }
- Include category and goal (select id, title, progress, targetValue, currentValue)
- Return null if not found

**`update(userId, id, data: UpdateTodoInput)`**
- Verify ownership: findFirst where { id, userId }, throw "Todo not found" if missing
- Convert date strings to Date objects
- prisma.todo.update with the data
- Return updated todo

**`delete(userId, id)`**
- Verify ownership, throw if missing
- prisma.todo.delete where { id }
- Return deleted todo

**`complete(userId, id)`** (the critical method with side effects)
1. Find the todo (include goal relation). Throw "Todo not found" if missing.
2. If already DONE, throw "Todo already completed".
3. Update the todo: `status: "DONE"`, `completedAt: new Date()`.
4. Award XP: call `gamificationService.awardXp(userId, id, "TODO", todo.priority)`. Note: the gamification service uses the horizon param to look up XP_PER_HORIZON. For to-dos, pass "TODO" as the horizon. This requires a small modification: in the awardXp call, use `XP_PER_TODO[todo.priority]` directly. Actually, looking at the gamification service, it takes goalId not todoId. The simplest approach: create the XpEvent manually in this method instead of calling awardXp, since the gamification service is goal-centric. Do this:
   - Calculate XP: `XP_PER_TODO[todo.priority] ?? 10`
   - Create XpEvent: `prisma.xpEvent.create({ data: { userId, amount: xp, source: "todo_complete:" + todo.priority, goalId: todo.goalId } })`
   - Upsert UserStats: increment totalXp by xp amount, handle weeklyScore reset same as gamification-service does
   - For simplicity, use prisma.userStats.upsert following the same pattern in gamification-service.ts
5. **Auto-increment linked goal progress (TODO-06):** If `todo.goalId` is set:
   - Call `goalService.logProgress(userId, todo.goalId, { value: 1, note: "Completed to-do: " + todo.title })`
   - This increments the goal's currentValue by 1 and recalculates progress percentage. If the goal has targetValue=10 and currentValue was 3, it becomes 4 (40%).
6. Return the completed todo with `_xp: { amount, source }` metadata.

**`skip(userId, id)`**
1. Find the todo, throw if missing.
2. If already SKIPPED, throw "Todo already skipped".
3. Update: `status: "SKIPPED"`, `completedAt: new Date()`.
4. No XP awarded, no goal progress increment.
5. Return the updated todo.

**`search(userId, query)`**
- Query with OR: title contains (insensitive), description contains (insensitive)
- Order by sortOrder asc, createdAt desc
- Include category and goal (select id, title)

The complete() method is the most important. The auto-increment of goal progress is the key feature linking inputs to outputs.

1. Calls `validateApiKey(request)` and returns `unauthorizedResponse()` if failed
2. Wraps logic in try/catch returning `handleApiError(error)` on failure
3. Parses input with Zod schemas
4. Calls todoService methods
5. Returns NextResponse.json()

**app/api/todos/route.ts** (GET + POST):
- GET: Parse query params into todoFiltersSchema. Call `todoService.list(auth.userId, filters)`. Return JSON array.
- POST: Parse body with `createTodoSchema.parse(body)`. Call `todoService.create(auth.userId, data)`. Return 201.

**app/api/todos/[id]/route.ts** (GET + PATCH + DELETE):
- Follow the exact same pattern as `app/api/goals/[id]/route.ts`
- GET: `todoService.getById`, return 404 if null
- PATCH: Parse body with `updateTodoSchema.parse(body)`, call `todoService.update`
- DELETE: `todoService.delete`
- The `params` destructuring uses `{ params }: { params: Promise<{ id: string }> }` (Next.js 16 async params)

**app/api/todos/[id]/complete/route.ts** (POST only):
- Call `todoService.complete(auth.userId, id)`
- Return the result (includes _xp metadata)
- This is a dedicated endpoint because completion has side effects (XP, goal progress), and keeping it separate from PATCH makes the API semantics clear

**app/api/todos/[id]/skip/route.ts** (POST only):
- Call `todoService.skip(auth.userId, id)`
- Return the result
