**1. Create `lib/services/dashboard-service.ts`:**

Export a `dashboardService` object with a single `getDashboardData(userId: string)` method that returns a `DashboardData` interface. The method runs 4 Prisma queries in parallel via `Promise.all`:

a) **weeklyFocus:** Find up to 5 goals where `userId`, `horizon: "WEEKLY"`, `status: { in: ["NOT_STARTED", "IN_PROGRESS"] }`, ordered by `priority: "desc"` then `deadline: "asc"`. Include `category`. After fetching, sort in JavaScript using a `PRIORITY_ORDER` map (`{ HIGH: 2, MEDIUM: 1, LOW: 0 }`) to guarantee correct priority ordering regardless of PostgreSQL enum ordinal. This addresses the Prisma enum sort pitfall from research.

b) **upcomingDeadlines:** Find goals where `userId`, `deadline: { gte: now, lte: addDays(now, 14) }`, `status: { in: ["NOT_STARTED", "IN_PROGRESS"] }`, ordered by `deadline: "asc"`. Include `category`. Use `date-fns` `addDays` for the 14-day window.

c) **allGoalsForCategoryAggregation:** Find all goals where `userId` and `categoryId: { not: null }`, selecting only `id`, `status`, `categoryId`, and `category: { select: { id, name, color, icon } }`. Aggregate in JavaScript using a Map to compute completed vs total per category, producing a `progressOverview` array with `{ categoryId, name, color, icon, total, completed, percentage }`.

d) **completedThisMonth:** Count goals where `userId`, `status: "COMPLETED"`, `completedAt: { gte: startOfMonth(now), lte: endOfMonth(now) }`. Use `date-fns` `startOfMonth` and `endOfMonth`.

After the parallel queries, run two more counts (totalGoals and totalCompleted for the user) to compute `completionRate`.

Also add a `getChildrenProgress(userId: string, goalId: string)` helper method that fetches children of a goal and computes the percentage of completed children. This is a read-time computation (PROG-05).

Define and export these TypeScript interfaces:
```typescript
export interface WeeklyFocusGoal { id: string; title: string; status: string; priority: string; progress: number; deadline: string | null; category: { id: string; name: string; color: string; icon: string | null } | null; }
export interface CategoryProgress { categoryId: string; name: string; color: string; icon: string | null; total: number; completed: number; percentage: number; }
export interface StatsData { completedThisMonth: number; totalGoals: number; totalCompleted: number; completionRate: number; currentXp: number; level: number; currentStreak: number; }
export interface DeadlineGoal { id: string; title: string; status: string; priority: string; horizon: string; deadline: string; category: { id: string; name: string; color: string; icon: string | null } | null; }
export interface DashboardData { weeklyFocus: WeeklyFocusGoal[]; progressOverview: CategoryProgress[]; streaksStats: StatsData; upcomingDeadlines: DeadlineGoal[]; }
```

For `streaksStats`, try to fetch the `UserStats` record for the user. If it exists, use its `totalXp`, `level`, `currentStreak`. If not (null), default to `{ currentXp: 0, level: 1, currentStreak: 0 }`. This avoids crashing when UserStats is not yet populated (Phase 9 concern from research).

Export three hooks:

a) `useDashboard()`: Uses `useQuery` with `queryKey: queryKeys.dashboard()` and `queryFn` that calls `fetchJson<DashboardData>("/api/dashboard")`. Import `DashboardData` type from the dashboard service.

b) `useLogProgress()`: Uses `useMutation` with `mutationFn` that takes `{ goalId: string; data: AddProgressInput }` and POSTs to `/api/goals/${goalId}/progress`. c) `useProgressHistory(goalId: string)`: Uses `useQuery` with `queryKey: queryKeys.goals.progress(goalId)` and `queryFn` that GETs `/api/goals/${goalId}/progress`. Set `enabled: !!goalId`.

All three hooks marked `"use client"` at the top.
