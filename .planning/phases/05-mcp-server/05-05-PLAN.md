**1. Create `lib/mcp/tools/category-tools.ts`:**

Export a `handleCategoryTool` function with the standard signature.

**create_category:**
- Parse args with `createCategorySchema.parse(args)`
- Call `categoryService.create(userId, parsed)`
- Return the created category

**update_category:**
- Extract `id` from args, validate non-empty string
- Parse remaining args with `updateCategorySchema.parse()` (remove `id` first)
- Call `categoryService.update(userId, id, parsed)`
- Return the updated category

**delete_category:**
- Extract `id`, validate non-empty string
- Call `categoryService.delete(userId, id)`
- Return confirmation: `"Category deleted successfully."`

**list_categories:**
- Call `categoryService.listTree(userId)` to get the nested tree structure
- Format as indented tree:
  ```
  "## Categories\n\n" +
  categories.map(c => formatCategory(c, 0)).join("\n")
  ```
  Where `formatCategory` renders: `"  ".repeat(indent) + "- " + name + " (" + color + ")" + children.map(...)join("\n")`
- Append raw JSON

Standard try/catch error handling for all cases.

**2. Create `lib/mcp/tools/data-tools.ts`:**

Export a `handleDataTool` function with the standard signature.

**export_data:**
- Extract `format` from args (default to "JSON")
- Fetch all data: `goalService.list(userId)` and `categoryService.list(userId)` in parallel via Promise.all
- Format based on `format`:

  **JSON:** `JSON.stringify({ exportedAt: new Date().toISOString(), goals, categories }, null, 2)`

  **CSV:** Build CSV with headers: `id,title,horizon,status,priority,progress,targetValue,currentValue,unit,deadline,categoryId,parentId,createdAt`. Map each goal to a CSV row. Handle commas and quotes in values by wrapping fields containing commas in double quotes and escaping internal double quotes. Prepend with a UTF-8 BOM for Excel compatibility is not needed here since this is text content, not a file download.

  **MARKDOWN:** Build a Markdown document with sections per horizon:
  ```
  # Ascend Goal Export\n\n
  *Exported: ${date}*\n\n
  ## Yearly Goals\n\n
  ${yearly goals as markdown checklist}\n\n
  ## Quarterly Goals\n\n
  ... etc for each horizon level
  ```
  Each goal: `- [${status === 'COMPLETED' ? 'x' : ' '}] **${title}** (${priority}) ${progress}%`

Return the formatted content as text.

**import_data:**
- Extract `data` string and `format` from args
- Parse the JSON string: `JSON.parse(data)`
- Validate the parsed structure has `goals` array and optionally `categories` array
- For the **standard JSON format**: iterate categories first (create each via `categoryService.create`), building an `oldId -> newId` mapping. Then iterate goals, creating each via `goalService.create`, mapping `categoryId` and `parentId` through the ID mappings. Process goals in order: YEARLY first, then QUARTERLY, then MONTHLY, then WEEKLY, so parent IDs are available.
- For **todos.json migration**: detect the old format by checking for fields like `tasks` or `projects`. If detected, map the old structure to the new goal format:
  - Old categories/projects become categories
  - Old tasks become WEEKLY goals
  - If the old format has a hierarchy, map to appropriate horizons
- Return a summary: `"Imported ${categoriesCreated} categories and ${goalsCreated} goals.\n\nDetails:\n${JSON.stringify(summary, null, 2)}"`
- If any individual import fails, skip it and collect errors. Return both successes and failures.

**get_settings:**
- For v1, settings are minimal. Since there is no UserSettings table in the schema yet, return sensible defaults:
  ```
  { theme: "system", defaultView: "list", apiKeyConfigured: true }
  ```
- Return as JSON: `"## Settings\n\n" + JSON.stringify(settings, null, 2)`

**update_settings:**
- For v1, since there is no settings persistence layer, return a message acknowledging the request but explaining that settings persistence is not yet implemented:
  ```
  "Settings update acknowledged. Note: Server-side settings persistence will be available in a future update. Current settings are managed client-side.\n\nRequested changes:\n" + JSON.stringify(args, null, 2)
  ```
- This is a pragmatic approach: the MCP tool exists (satisfying MCP-20), but the actual persistence will come when a UserSettings model is added.

Standard try/catch error handling for all cases.
