**Create `app/api/mcp/route.ts`:**

This is the critical adapter between Next.js App Router (Web Standard Request/Response) and the MCP SDK's Server class. Since the server is stateless (no SSE, no sessions), implement a manual JSON-RPC bridge that avoids the `StreamableHTTPServerTransport` entirely (which expects Node.js IncomingMessage/ServerResponse).

Export three route handlers: `POST`, `GET`, `DELETE`.

**POST handler (main MCP endpoint):**

```typescript
export async function POST(request: NextRequest) {
  // 1. Authenticate FIRST (before any MCP processing)
  const auth = await validateApiKey(request);
  if (!auth.success) {
    return NextResponse.json(
      { jsonrpc: "2.0", error: { code: -32000, message: "Unauthorized" }, id: null },
      { status: 401 }
    );
  }

  // 2. Parse JSON-RPC request body
  const body = await request.json();

  // 3. Create server instance scoped to this user
  const server = createAscendMcpServer(auth.userId);

  // 4. Handle the JSON-RPC request by calling the server's internal handler
  // The Server class exposes a method to handle requests programmatically.
  // Use server's transport connection with an in-memory approach.
}
```

**The transport bridge pattern:**

Since the SDK Server requires a transport to process messages, create a minimal in-memory transport:

1. Create a `StreamableHTTPServerTransport` with `sessionIdGenerator: undefined` (stateless mode).
2. Connect the server to this transport via `await server.connect(transport)`.
3. The transport needs to receive the parsed JSON-RPC body and return a response.

**Alternative (simpler) approach if the transport bridge is complex:**

Use the `StreamableHTTPServerTransport`'s `handleRequest` method if available, or use the low-level approach of:

1. Creating a `ReadableStream` from the request body
2. Creating a `WritableStream` to collect the response
3. Piping through the transport

After reviewing the SDK, the practical approach is to use `StreamableHTTPServerTransport` and pipe the request/response through it. The transport handles JSON-RPC framing.

Create a helper function `handleMcpRequest(server: Server, body: unknown): Promise<unknown>` that:

1. Creates a `StreamableHTTPServerTransport` with `{ sessionIdGenerator: undefined }` for stateless mode
2. Connects the server to the transport
3. Sends the body through the transport
4. Collects and returns the response

If the SDK's transport does not support direct body-in/response-out (it expects HTTP request/response objects), fall back to this pattern:

Create a simple request handler that matches the JSON-RPC method to the server's registered handlers:
- For `initialize`: return server info and capabilities
- For `tools/list`: call the ListToolsRequestSchema handler
- For `tools/call`: call the CallToolRequestSchema handler
- For `ping`: return `{ jsonrpc: "2.0", result: {}, id: body.id }`

This manual dispatch is the recommended path from the research (Pattern 3, Option 1: manual JSON-RPC handling for stateless POST-only servers).

Implement the JSON-RPC dispatcher:

```typescript
async function handleJsonRpc(server: Server, userId: string, body: any) {
  const { method, params, id } = body;

  switch (method) {
    case "initialize":
      return {
        jsonrpc: "2.0",
        result: {
          protocolVersion: "2025-03-26",
          capabilities: { tools: {} },
          serverInfo: { name: "ascend", version: "1.0.0" },
        },
        id,
      };

    case "notifications/initialized":
      // Client acknowledgment, no response needed for notifications
      return null;

    case "ping":
      return { jsonrpc: "2.0", result: {}, id };

    case "tools/list":
      // Delegate to server's registered handler
      const tools = await server.getRequestHandler(ListToolsRequestSchema)?.(params);
      return { jsonrpc: "2.0", result: tools, id };

    case "tools/call":
      const result = await server.getRequestHandler(CallToolRequestSchema)?.(params);
      return { jsonrpc: "2.0", result, id };

    default:
      return {
        jsonrpc: "2.0",
        error: { code: -32601, message: `Method not found: ${method}` },
        id,
      };
  }
}
```

**IMPORTANT:** The Server class's handler access pattern. Check the SDK source to determine how to invoke registered handlers. The `setRequestHandler` stores handlers internally. The `Server` class from `@modelcontextprotocol/sdk/server/index.js` has a method to dispatch requests. If direct handler invocation is not public API, use the transport approach instead.

**Fallback: Use the transport with a custom adapter.** If direct handler access is blocked, create a pair of `ReadableStream`/`WritableStream` to pipe through the transport. The key insight is that for stateless mode, each request creates a fresh server+transport, processes one message, and returns.

**GET handler:** Return 405 Method Not Allowed with JSON body `{ error: "Method not allowed. Use POST for MCP requests." }`.

**DELETE handler:** Return 405 Method Not Allowed with JSON body `{ error: "Method not allowed. This is a stateless MCP server." }`.

Add the `Content-Type: application/json` header to all responses. For the POST handler, also set `Content-Type: application/json` (not `text/event-stream` since this is stateless, not SSE).
